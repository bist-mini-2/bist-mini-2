# 이미지/파일 다운로드 API 구현 방식 비교 (Header + JS vs Query String + Browser)

파일이나 이미지를 다운로드하거나 렌더링하는 API를 설계할 때, 인증 토큰(Access Token)을 전달하고 처리하는 두 가지 주요 방식을 비교한 가이드입니다.


---


## **1. JavaScript(Fetch/Axios) + HTTP Header 방식**

자바스크립트 비동기 요청을 사용해 API를 호출하고 헤더에 토큰을 실어 응답 데이터를 받아온 후, 이를 Blob 객체로 변환하여 다운로드를 유도하는 방식입니다.


### **① 동작 흐름**

1. 프론트엔드에서 `fetch` 또는 `axios`로 API 호출 (헤더에 `Authorization: Bearer <Token>` 설정).
1. 서버에서 파일 바이너리를 스트림/버퍼로 응답.
1. 프론트엔드는 응답 데이터를 메모리에 `Blob`으로 로드.
1. `URL.createObjectURL(blob)`을 통해 브라우저 임시 메모리 URL 생성.
1. 가상의 `<a href="blob:..." download="파일명">` 태그를 동적 생성 및 클릭 이벤트를 트리거하여 로컬에 저장.
1. 다운로드 완료 후 `URL.revokeObjectURL`을 호출해 생성했던 메모리 해제.

### **② 장단점**

* **장점**:
  * **보안성 극대화**: 토큰이 HTTP 요청 헤더에 캡슐화되므로 URL에 절대 노출되지 않으며 서버 로그, 브라우저 히스토리 등에 흔적이 남지 않습니다.
  * **일관성**: 기존의 모든 API 통신(인터셉터, 공통 헤더 처리)과 동일한 보안 패턴을 사용할 수 있습니다.
* **단점**:
  * **메모리 부족(OOM) 위험**: 전체 파일 바이너리를 먼저 브라우저 메모리(Blob)에 전부 올린 후에 다운로드가 시작되므로, **수백 MB 이상의 대용량 파일 다운로드 시 브라우저 탭이 다운(Crash)**될 수 있습니다.
  * **구현 복잡도**: 가상 태그 생성, Blob 변환 및 해제 등 프론트엔드 추가 스크립트 작성이 필수적입니다.

## **2. 브라우저 내장 태그 (****`<a href="...">`****) + Query String 방식**

인증용 토큰을 URL의 쿼리 스트링에 포함시키고, 브라우저가 기본적으로 지원하는 파일 다운로드 흐름을 직접 타게 만드는 방식입니다.


### **① 동작 흐름**

1. HTML에 `<a href="/api/download?accessToken=Token" download>` 태그 작성.
1. 사용자가 클릭 시 브라우저가 직접 해당 URL로 GET 요청을 송신 (자바스크립트를 거치지 않음).
1. 서버는 응답 헤더에 `Content-Disposition: attachment; filename="..."`를 설정하여 전송.
1. 브라우저가 직접 파일 저장 다이얼로그를 표시하며 로컬 디스크에 저장 시작.

### **② 장단점**

* **장점**:
  * **대용량 파일 스트리밍 최적화**: 브라우저가 데이터를 수신하는 대로 실시간으로 하드디스크에 직접 쓰기 때문에 브라우저 자체의 메모리를 거의 차지하지 않습니다. **기가바이트 단위의 초대용량 다운로드도 매우 안정적**입니다.
  * **단순성**: 프론트엔드 코드 작성이 극히 단순해지며 브라우저의 기본 다운로드 기능(속도 표시, 이어받기 등)을 그대로 활용할 수 있습니다.
* **단점**:
  * **보안 취약성**: 토큰이 URL 쿼리 파라미터로 노출되므로, 프록시/서버 로그에 남거나 외부 참조 시 Referer 헤더를 통해 유출될 수 있습니다.
  * **대응책**: 보안 취약점을 막기 위해 파일 다운로드용으로 수 초 동안만 쓸 수 있는 **단기 만료 일회성 토큰 (One-Time Token / Signed URL)** 정책을 함께 사용해야 합니다.

## **3. 서버 API를 '스트리밍 다운로드(FastAPI StreamingResponse 등)'로 구현한 경우의 영향**

서버에서 대용량 파일 전송을 위해 데이터를 청크(Chunk, 조각) 단위로 나누어 전송하는 **스트리밍 방식(StreamingResponse)**으로 API를 구현했더라도, **프론트엔드(클라이언트)가 이를 어떻게 호출하느냐**에 따라 효과가 완전히 다릅니다.


### **① Header + JS (Fetch/Axios)로 호출하는 경우**

* **한계**: 서버가 데이터를 조각 단위로 전송(Streaming)하더라도, 일반적인 JavaScript HTTP 클라이언트(`fetch`, `axios`)는 **전체 스트림이 종료될 때까지 기다린 후 최종 바이너리를 반환**하는 방식으로 작동합니다.
* 결국 자바스크립트 코드 내에서 `response.blob()` 등을 호출하여 메모리에 적재하는 순간, 모든 청크가 하나의 커다란 메모리 공간에 모이기 때문에 **메모리 부족(OOM) 크래시 현상은 전혀 해결되지 않습니다.**
* **JS로 진짜 스트리밍 저장을 하려면**:
  * 브라우저의 `ReadableStream` API(`response.body.getReader()`)를 직접 열어 들어오는 청크를 실시간으로 받아내고, 이를 **File System Access API** 등의 `WritableStream`을 이용해 하드디스크 파일에 직접 한 조각씩 바로 써야 합니다.
  * 하지만 이 방식은 최신 브라우저 위주로만 제한적 지원되며 보안 다이얼로그 확인 등 프론트엔드 코드가 대단히 복잡해집니다.

### **② 브라우저 내장 태그 (****`<a href="...">`****) + Query String으로 호출하는 경우**

* **시너지 효과**: 서버의 스트리밍 응답과 브라우저의 내장 다운로드 엔진이 완벽한 조화를 이룹니다.
* 브라우저의 다운로드 매니저는 서버로부터 유입되는 데이터 청크를 **중간에 메모리에 올리지 않고 바로 지정된 로컬 디스크 경로에 스트리밍으로 기록**합니다.
* 따라서 서버가 스트리밍 다운로드 API로 최적화되어 있을 때, 그 성능적 이점(메모리 절약, 안정성)을 제대로 누리기 위해서는 **브라우저 내장 태그 방식(Query String 방식)**으로 호출하는 것이 가장 바람직한 방법입니다.

## **4. 보안성과 성능(스트리밍 디스크 저장)을 동시에 충족하는 3가지 실무 패턴**

보안상 메인 Access Token을 URL에 노출하지 않으면서도(Header 인증처럼 안전하게), 대용량 파일 다운로드 시 브라우저 OOM 크래시를 피하기 위해(브라우저 내장 다운로드 엔진 활용) 실무에서 널리 쓰이는 아키텍처 패턴들입니다.


### **① 일회성 토큰 (One-Time Token / Signed URL) 패턴 (가장 추천)**

메인 인증(JWT)을 통해 아주 짧은 유효 기간(예: 10초 ~ 30초)을 가진 임시 티켓(One-Time Token)을 먼저 발급받은 뒤, 그 임시 티켓을 쿼리 스트링에 실어 브라우저 내장 다운로드 기능을 타게 하는 하이브리드 방식입니다.

* **동작 흐름**:
  1. 사용자가 다운로드 버튼을 클릭하면, 자바스크립트가 기존 HTTP Header(`Authorization`) 인증을 사용하여 서버에 **"다운로드 티켓 발급 API"**를 호출합니다.
  1. 서버는 본 파일 정보와 연동된 임시 랜덤 토큰(OTT)을 생성하여 DB나 Redis에 저장(수명 10초)하고, 프론트엔드에 이 토큰이 포함된 임시 다운로드 URL을 응답합니다.
    * 예: `/api/download?ticket=temp_random_token_xyz`
  1. 프론트엔드는 응답받은 URL을 이용해 브라우저 내장 다운로드 태그(`<a href="...">`)를 동적으로 생성 및 클릭하여 실행합니다.
  1. 다운로드 API 서버는 요청이 유입되면 쿼리의 `ticket` 값을 Redis 등에서 조회해 검증하고, **검증 즉시 해당 티켓을 소멸(1회성)** 시킨 후 대용량 스트리밍 파일 전송을 진행합니다.
* **장점**:
  * 메인 Access Token이 로그에 남지 않아 절대 안전합니다.
  * 임시 티켓은 10초 내에만 유효하며 한 번 다운로드가 개시되면 파기되므로 외부 노출 위험이 매우 낮습니다.
  * 디스크 스트리밍 저장이 작동하여 메모리 효율이 극대화됩니다.
* **단점 및 고려사항**:
  * **인프라 복잡도 증가 (상태 관리 필요)**: 임시 발급한 티켓의 사용 상태를 기록하고 검증 즉시 소멸시켜야 하므로 Redis나 데이터베이스 같은 상태(State) 저장소가 필요합니다. 무상태(Stateless) 아키텍처를 추구하는 JWT의 장점이 다소 희석됩니다.
  * **네트워크 왕복(Round Trip) 증가**: 사용자가 클릭했을 때 [1단계: 티켓 발급 API 호출 ➔ 2단계: 실제 다운로드 URL 요청] 순으로 총 2번의 네트워크 요청이 발생하므로 레이턴시가 미세하게 증가합니다.
  * **다운로드 이어받기(Resume) 및 재시도 제약**: 다운로드가 중간에 끊겨 브라우저가 다시 시도(Retry)하거나 이어받기(Range Request)를 요청할 때, 이미 사용되어 소멸된 1회성 티켓으로는 다운로드를 재개할 수 없습니다. (상황에 따라 유효시간을 1~2분 정도로 여유 있게 설정해야 합니다.)

### **② HttpOnly 쿠키 기반 인증 패턴**

토큰을 Authorization 헤더 대신 브라우저의 쿠키 영역에 저장하고, 브라우저가 직접 요청할 때도 쿠키를 통해 인증하도록 하는 방식입니다.

* **동작 흐름**:
  1. 로그인 시 서버가 `Access Token`을 HTTP Response Header의 `Set-Cookie` 속성(`HttpOnly`, `Secure`, `SameSite=Lax/Strict`)을 부여하여 전달합니다.
  1. 브라우저는 자바스크립트 없이 HTML 내장 태그(`<img src="/api/images/1">`, `<a href="/api/download">`)로 직접 요청하더라도 동일 도메인(SameSite) 내 요청이라면 **자동으로 쿠키를 HTTP 요청 헤더에 담아서 전송**합니다.
  1. 서버는 들어온 쿠키 정보를 검증하여 파일을 스트리밍 응답합니다.
* **장점**:
  * 자바스크립트(`XSS` 공격)로 쿠키 탈취가 불가능해 보안성이 매우 우수합니다.
  * URL에 토큰 관련 쿼리 파라미터가 보이지 않으므로 서버 액세스 로그나 브라우저 히스토리에 토큰이 노출될 염려가 전혀 없습니다.
  * 브라우저 기본 태그 방식을 그대로 쓰므로 완벽한 디스크 스트리밍이 보장됩니다.
* **단점**:
  * CSRF(Cross-Site Request Forgery) 공격에 대한 추가 방어가 요구됩니다.
  * 타 도메인(Cross-Origin) 환경(예: API 서버와 프론트엔드 도메인이 전혀 다른 경우)에서는 쿠키 공유 설정이 매우 까다로울 수 있습니다.

### **③ Service Worker (서비스 워커) 중계 패턴**

자바스크립트의 서비스 워커를 백그라운드에 등록하여, 브라우저 내장 태그의 요청을 서비스 워커가 중간에서 가로채 헤더를 붙여 재요청을 보내는 방식입니다. (예: `StreamSaver.js` 라이브러리 활용)

* **동작 흐름**:
  1. 클라이언트단에 서비스 워커를 사전 등록합니다.
  1. 다운로드 요청 시 서비스 워커가 해당 파일 다운로드 요청을 인터셉트(Intercept)합니다.
  1. 서비스 워커는 메모리에 보관 중이던 Access Token을 헤더(`Authorization`)에 부착하여 서버에 파일 스트림을 요청합니다.
  1. 서버의 응답 스트림이 들어오면 서비스 워커가 이를 브라우저의 쓰기 가능 스트림(`WritableStream`)으로 파이프라이닝(Pipelining)하여 사용자의 하드디스크에 실시간으로 써 내려갑니다.
* **장점**:
  * URL에 토큰 노출이 전혀 없으며, 완벽한 자바스크립트 제어 하에 있으면서도 OOM 크래시 없이 스트리밍 다운로드가 가능합니다.
* **단점**:
  * 구현 난이도가 최상급이며, 모바일 브라우저 환경 등 일부 노후 브라우저에서 서비스 워커 호환성에 제약이 있을 수 있습니다.

## **5. 요약 비교표**

| 비교 항목 | ① Header + JS 방식 | ② Query String + Browser 방식 | ③ 일회성 토큰 (OTT) 패턴 (추천) | ④ HttpOnly 쿠키 방식 |
| --- | --- | --- | --- | --- |
| **요청 헤더 제어** | 가능 | 불가능 | **티켓 발급 단계에서 가능** | 불가능 (브라우저 자동 전송) |
| **토큰 전달 경로** | HTTP Request Header | URL Query Parameter | **임시 티켓을 쿼리로 전달** | HTTP Request Cookie |
| **브라우저 메모리 사용** | **높음** (OOM 발생 가능) | **낮음** (실시간 디스크 기록) | **낮음** (실시간 디스크 기록) | **낮음** (실시간 디스크 기록) |
| **대용량 파일 지원** | 불가 | **최적** | **최적** | **최적** |
| **보안성 (토큰 노출)** | 우수 (노출 없음) | 취약 (URL에 토큰 노출) | **우수** (임시 티켓만 노출) | **우수** (URL 노출 없으며 XSS 안전) |
| **구현 난이도** | 보통 | 매우 쉬움 | **보통** (임시 API/저장소 필요) | 보통 (SameSite 등 쿠키 설정 필요) |
