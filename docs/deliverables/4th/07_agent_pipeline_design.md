## 7-1. 오케스트레이션 도입의 기술적 배경과 성과

단일 거대언어모델(LLM) 호출에 의존하는 일반적인 RAG 시스템은 복잡한 다중 지식 소스를 조인할 때 프롬프트가 거대해져 핵심 지시문(Instruction)이 누수되거나, 관련성이 낮은 노이즈 정보가 유입되어 환각(Hallucination) 현상이 증폭되는 기술적 한계에 부딪혔습니다.

본 플랫폼은 이를 해결하기 위해 **LangGraph 기반의 상태 전이(State Transition) 오케스트레이션**을 도입 및 구현했습니다. 각 연산 단계(질의 분석 ➡️ 문서/웹 탐색 ➡️ 보고서 합성 ➡️ DB 백업)를 독립적인 에이전트 노드로 완전 격리하고, 런타임 상태를 중앙 스토어인 `MultiAgentState`를 통해 투명하게 전이되도록 설계했습니다. 그 결과, 지식 합성의 엄밀성을 엔지니어링 수준에서 강력하게 통제할 수 있게 되었으며, 단일 모델의 인지적 과부하를 고르게 분산하여 답변의 정확성을 극대화하는 성과를 달성했습니다.

## 7-2. 일반 챗 허브 (Unconditional Parallel Execution)

일반 챗 허브는 답변 생성 속도의 획기적인 향상을 위해 데이터 라우팅 분기 판단 단계를 생략하고, **학술 RAG와 실시간 웹 검색을 무조건적으로 동시에 가동(Unconditional Parallel Execution)**하여 응답 지연을 방지하고 정보의 풍부함을 극대화했습니다.

> *   **업로드 대상 파일**: `docs/deliverables/4th/ai_agent_workflow.png` (가로 폭에 맞추어 리사이즈 권장)

### 1) asyncio.gather 기반 병렬 처리 및 레이턴시 단축 성과
*   **병목 구간 해소**: 기존의 순차 라우팅 및 순차 실행 방식(의도 분석 ➡️ 논문 RAG ➡️ 웹 검색 ➡️ 답변 합성)은 각 I/O 대기 시간이 누적되어 평균 **9.22초**의 지연 시간이 발생했습니다.
*   **비동기 최적화 성과**: 본 플랫폼은 `asyncio.gather` 비동기 태스크 큐를 활용하여 pgvector 코사인 유사도 검색(`PaperNode`)과 Tavily Search API 웹 크롤링(`WebNode`)을 무조건적으로 병렬 가동했습니다. 이로 인해 총 응답 지연 시간이 **7.10초로 단축(약 2.12초, 23% 성능 향상)**되었으며, 실시간으로 풍부한 지식을 화면에 스트리밍(SSE)하는 쾌적한 UX를 완성했습니다.

## 7-3. Gem 팩토리: 클로저(Closure) 기반 동적 도구 바인딩 및 보안 격리

맞춤형 비서인 Gem 팩토리는 사용자가 생성한 젬(Gem)의 파일 데이터 셋을 엄격히 격리하면서도, 런타임에 유연하게 동작하는 프롬프트 바인딩 및 보안 파쇄 기술을 구축했습니다.

> 📢 **[구글 독스 이미지 삽입 영역]**
> *   구글 독스 메뉴의 `삽입 ➡️ 이미지 ➡️ 컴퓨터에서 업로드`를 클릭한 뒤, 아래 파일을 선택해 삽입해 주세요:
> *   **업로드 대상 파일**: `docs/deliverables/4th/system_architecture_tier2.png`

### 1) 클로저(Closure) 기반 동적 툴 바인딩 구현
*   **기술적 구현**: 젬 대화가 가동되는 런타임 시점에, 해당 젬의 고유 식별자(`gem_id`)를 내부 메모리 영역에 격리하여 캡처하는 **클로저(Closure) 함수**를 백엔드에서 실시간 생성해 Agent Executor에 동적으로 주입(Dynamic Binding)했습니다.
*   **클로저 구현 메커니즘 (코드 요약)**:
    *   `def create_gem_search_tool(gem_id: str):`
    *   &nbsp;&nbsp;&nbsp;&nbsp;`async def gem_search_tool(query: str):`
    *   &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;`return await RAG.search_isolated(collection=f"gem_{gem_id}_files", query=query)`
    *   &nbsp;&nbsp;&nbsp;&nbsp;`return gem_search_tool`
*   **보안 성과**: 젬 비서가 런타임 상에서 오직 자신에게 지정된 `gem_{gem_id}_files` 테이블 컬렉션에서만 탐색하도록 강제함으로써, 타인이나 다른 대화방으로의 정보 유출을 구조적·원천적으로 차단했습니다.

### 2) 데이터 영구 소멸 (Wipe-out) 자동화 수립
*   **Wipe-out 메커니즘**: 사용자가 젬을 스토어에서 영구 소멸시키거나 세션을 종료하는 시점에, 관계형 메타 소거와 동시에 pgvector 물리 레코드의 `DROP COLLECTION` 쿼리를 단일 트랜잭션으로 연동 처리하는 완전 소거 프로세스를 구현했습니다.
*   **보안 성과**: 임시 파싱 파일이나 벡터 임베딩 데이터가 서버 스토리지에 어떠한 잔여 용량(Byte)도 남기지 않고 디스크 레벨에서 완전 소멸(0 Byte)되는 높은 수준의 기밀성을 확보했습니다.
