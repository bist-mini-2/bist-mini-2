# 7. 핵심 기능별 에이전트 파이프라인 설계 및 최적화 성과

본 문서는 `bist-mini-2` 플랫폼의 핵심인 AI 오케스트레이션을 **단방향 요청 처리 파이프라인 아키텍처 흐름(Request Lifecycle Pipeline Flow)** 기준으로 재구성하여 기술한 프로젝트 최종 완성 보고서(Completion Report)입니다. 데이터의 유입(Ingress)부터 인출, 융합, 스트리밍, 사후 영구 삭제에 이르는 데이터 저니(Data Journey)를 단계별로 명확히 기술합니다.

## 7-2. 파이프라인 아키텍처 단계별 세부 기술 명세

### [Phase 1] 게이트웨이 요청 진입 및 의도 분석 (Ingress & Analysis)
*   **작동 메커니즘**: 클라이언트의 자연어 질문이나 맞춤형 연구 비서(Gem) 요청이 유입되면, 파이프라인의 입구인 `AnalysisNode`가 즉각 가동됩니다.
*   **상세 기술**: `gpt-4o-mini` 모델이 프롬프트 과대화를 예방하기 위해 자연어 질문에서 RAG에 적합한 핵심 영어 학술 키워드(`paper_query`)와 최신 상용 동향 수집용 검색어(`web_query`)를 분리·추출합니다.
*   **성과**: 질의 의도를 정형화된 메타 쿼리로 먼저 해체함으로써, 이후 단계인 검색 인출 레이어의 지식 순도와 정확도를 향상시킵니다.

### [Phase 2] 듀얼 트랙 병렬 검색 및 물리 데이터 격리 (Dual-Track Retrieval & Isolation)
*   **작동 메커니즘**: 의도 분석이 완료된 쿼리들을 타겟 데이터 소스 레이어로 전달하여 병렬 조회를 실행합니다.
*   **상세 기술**:
    1.  **일반 챗 허브**: pgvector 논문 DB 조회(`PaperNode`)와 Tavily 웹 크롤링(`WebNode`)을 `asyncio.gather` 비동기 라이브러리를 통해 무조건적으로 동시 가동합니다. 이를 통해 순차 라우팅 대비 연산 레이턴시를 **9.22초에서 7.10초로 약 23% 단축**하는 성과를 거두었습니다.
    2.  **젬 팩토리**: 런타임 세션이 생성되는 즉시, 해당 젬의 고유 식별자(`gem_id`)를 메모리 공간에 가두는 **클로저(Closure) 함수**를 동적으로 생성하여 툴을 주입합니다. 각 젬은 오직 자신에게 할당된 `gem_{gem_id}_files` pgvector 컬렉션 공간에서만 벡터 탐색을 수행하므로 물리적인 격리가 실현됩니다.

### [Phase 3] 다자 지식 크로스 융합 및 합성 (Cross-Synthesis & Orchestration)
*   **작동 메커니즘**: 수집 완료된 상이한 소스의 지식 조각들을 조인(Join)하여 하나의 학술 보고서로 조합하는 통합 단계입니다.
*   **상세 기술**: `SynthesisNode` (`gpt-4o` 기용)가 실행되어, DB에서 추출한 학술적 팩트 이론(RAG)과 실시간 웹 수집 정보를 단일 프롬프트 문맥 내에서 크로스 체크 및 합성합니다.
*   **상태 제어**: LangGraph의 `MultiAgentState` 스토어가 노드 간 매개 변수를 중재 및 투명하게 격리함으로써, 정보가 유실되거나 상태가 꼬이는 오류를 원천 예방합니다.

### [Phase 4] 실시간 토큰 스트리밍 전송 (SSE-based Delivery)
*   **작동 메커니즘**: 합성 노드에서 생성되는 텍스트를 클라이언트 브라우저에 지연 없이 밀어내는 아웃바운드(Egress) 단계입니다.
*   **상세 기술**: FastAPI의 `StreamingResponse`와 비동기 제너레이터를 연동하여, 합성되는 단어 토큰을 HTTP Server-Sent Events(SSE) 채널을 통해 즉각 전송합니다.
*   **사용자 경험 성과**: 전체 합성 연산이 끝날 때까지 브라우저 화면이 멈춰있지 않고, 첫 단어가 1초 내에 화면에 타이핑되듯 렌더링되어 최적의 UX를 보장합니다.

### [Phase 5] 사후 세션 영구 보존 및 물리 파쇄 (Post-Processing & Wipe-out)
*   **작동 메커니즘**: 트랜잭션이 성공적으로 완결된 이후 데이터의 뒤처리 및 메모리 가드를 총괄하는 사후 제어 단계입니다.
*   **상세 기술**:
    1.  **인용 출처 백업**: 사용된 ArXiv 논문 서지 정보를 관계형 DB의 `chat_source` 테이블에 비동기 Commit 적재하여 후속 대화방 진입 시 출처 카드를 보존 및 복원합니다.
    2.  **데이터 물리 소멸 (Wipe-out)**: 사용자가 젬을 스토어에서 영구 삭제 처리하면, 데이터베이스 관계형 메타 데이터가 Cascade 삭제됨과 동시에 pgvector 상의 물리 임베딩 격리 테이블 공간을 `DROP COLLECTION` 쿼리로 영구 드롭하여 디스크 내 잔여 용량(Byte)을 완전히 0으로 회수(Shredding)합니다.
