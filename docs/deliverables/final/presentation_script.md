저희 플랫폼의 핵심 비전은 **'연구자 및 R&D 센터를 위한 맞춤형 AI 공동 연구 파트너'**입니다. 파편화된 학술 데이터베이스와 시시각각 변화하는 실시간 웹 동향을 하나의 지능형 컨텍스트로 통합하여, 연구자가 단순 자료 조사에 소모하는 비용을 최소화하고 본연의 연구 활동에 집중할 수 있도록 돕는 것이 목표입니다.

역할 분담의 경우, 3대 핵심 도메인별 RAG 파이프라인 구축을 기반으로, 저는 컴퓨터 과학 분야 RAG와 대규모 문헌 분석 설계를, 신동원 님은 천문학 RAG와 사용자 정의 젬 팩토리를, 천승현 님은 생명공학 RAG와 채팅 허브 기능 개발을 전담하여 병렬적으로 속도감 있게 개발을 진행했습니다.

개발 과정은 총 2주의 타이트한 일정 속에서 진행되었습니다. 
* **Phase 1**인 첫 이틀 동안은 3대 분야 총 10만 6천여 건의 대용량 학술 데이터셋을 성공적으로 적재하여 기반 인프라를 완성했고, 
* **Phase 2**에서는 이를 활용해 실시간 채팅 허브, 비동기 분석기, 젬 팩토리 등의 핵심 비즈니스 로직을 구축했습니다. 
* 마지막 **Phase 3**에서는 통합 비동기 테스트와 API 검증을 완료하고 로컬 및 전체 배포 환경을 완성하였습니다.

그럼 이러한 서비스를 기획하게 된 비즈니스적 배경을 다음 장의 Lean Canvas를 통해 살펴보겠습니다."

학술 연구 현장에서 마주하는 **가장 큰 문제**는 매일 쏟아지는 논문을 조사하는 데 드는 피로감, 일반 LLM을 사용할 때 발생하는 환각 현상으로 인한 교차 검증 비용, 그리고 민감한 연구 데이터가 외부 LLM 서비스로 유출될지 모른다는 IP 유출 우려입니다. 
저희는 이러한 문제를 절실히 느끼는 **석·박사 연구원, R&D 센터 책임연구원, 그리고 특허 변리사** 분들을 핵심 타겟 고객으로 설정했습니다.

이에 대한 **솔루션**으로 저희는 학술 DB와 실시간 웹 검색을 병렬로 융합한 듀얼 트랙 RAG 스트리밍, 대규모 논문을 분석하는 연구 공백 분석기, 그리고 나만의 페르소나 연구 비서를 만드는 젬 팩토리를 제시합니다. 

특히 저희 플랫폼이 갖는 **독점적 우위**는 단순한 LLM 래핑이 아니라, 10만 건 이상의 데이터를 단 68ms만에 조회하는 pgvector HNSW 아키텍처와 한글 번역 시 원문 영문 코드를 100% 보존하는 번역 가드 필터 등의 고도화된 기술력에 있습니다.

수익 모델의 경우, 개인 연구자를 위한 월 19.99달러의 Pro 멤버십 구독 서비스뿐만 아니라, 보안이 극도로 중요한 기업 및 연구소를 타겟으로 한 **보안 On-premise 구축 패키지**까지 염두에 두고 비즈니스 모델을 설계했습니다.

그렇다면 이 비즈니스 로직을 실현하기 위한 시스템 아키텍처를 자세히 설명해 드리겠습니다."

#### 🎙️ 발표 대본
"플랫폼의 시스템 아키텍처와 데이터 관리 현황입니다.

저희 플랫폼은 견고하고 현대적인 **3-Tier 물리적 레이어**로 구성되어 있습니다.
* **프론트엔드**는 Next.js App Router를 기반으로 하여 빠른 렌더링 속도와 우수한 SEO를 확보했고, 실시간 대화를 위해 SSE(Server-Sent Events) 프로토콜을 Axios와 연동하여 자연스럽게 스트리밍 통신이 가능하도록 설계했습니다.
* **백엔드**는 비동기 ASGI 서버인 FastAPI를 채택하였으며, Pydantic v2 DTO를 활용해 데이터 유효성 검증을 원천화했습니다.
* **데이터베이스** 영역에서는 PostgreSQL 17 버전과 pgvector 플러그인을 결합하고, HNSW 인덱싱을 통해 3072차원의 고차원 벡터 임베딩 매칭을 평균 68ms라는 초고속 성능으로 응답하도록 설계했습니다.
* **AI 에이전트** 레이어는 LangChain과 LangGraph를 활용해 OpenAI의 프롬프트 흐름을 제어하고, Tavily Search API를 결합하여 필요한 경우 실시간 웹 정보까지 동시 수집하는 병렬 오케스트레이션을 완성했습니다.

우측을 보시면, 실제 적재된 **대용량 학술 데이터 현황**입니다.
저희는 생명과학, 천문학, 컴퓨터공학 3대 분야의 ArXiv 학술 데이터 총 106,974건을 수집해 적재했습니다.
생명과학 5만 4천여 건, 천문학 3만 5천여 건, 컴퓨터공학 1만 7천여 건이며, OpenAI의 3세대 대형 임베딩 모델(text-embedding-3-large)을 사용해 3,072차원의 의미 공간을 완벽하게 구축했습니다.

이제 이러한 아키텍처 위에서 구현된 핵심 기능들을 하나씩 살펴보겠습니다."

#### 🎙️ 발표 대본
"플랫폼의 가장 핵심적인 비즈니스 로직인 **'일반 채팅 허브'**와 **'연구 공백 분석기'**에 대해 설명드리겠습니다.

먼저 1번 **일반 채팅 허브**는 자연어 질문이 들어왔을 때, 내부 학술 데이터베이스 RAG와 실시간 웹 검색(Tavily)을 동시에 구동합니다.
* 이 과정에서 `asyncio.gather`를 활용한 **무조건적 병렬 인출** 방식을 구현하여, 순차적으로 데이터를 찾을 때보다 대기 지연 시간을 평균 23% 단축시켰습니다.
* 또한, 생성되는 답변은 HTTP SSE 방식을 채택하여 클라이언트 단에 글자 단위로 실시간 타이핑되듯이 렌더링되며,
* 답변에 인용된 모든 ArXiv 논문의 메타데이터는 1:1로 매핑되어 `chat_source` 테이블에 영구 적재되므로 연구자는 언제든지 신뢰성 있는 원문을 직접 추적하고 검증할 수 있습니다.

우측의 2번 **연구 공백 분석기**는 대량의 연구 논문을 다각도로 분석하여 아직 연구되지 않은 '미개척 가설'을 제안해 주는 기능입니다.
* 다수의 논문을 분석하는 작업은 시간이 오래 걸리므로, 백엔드의 `BackgroundTasks`로 오프로딩하여 비동기 처리하며, 사용자는 화면에서 진행률이 10%부터 100%까지 실시간으로 차오르는 인터랙티브한 UI를 경험할 수 있습니다.
* 영문 논문 요약 및 분석 결과를 한글로 번역하는 과정에서, 학술적으로 엄격하게 보존되어야 하는 영문 핵심 구절이나 특정 전문 용어가 왜곡되는 것을 막기 위해 파이썬 단에서 원어를 백업해 두었다가 번역 완료 후 복원하는 **'100% 팩트 보존 번역 가드'**를 적용했습니다.
* 최종 분석이 완료되면 SSE 완료 알림을 통해 사용자에게 즉시 토스트 팝업으로 알려주며, 연산 부하가 큰 분석 결과물은 Redis 캐시 및 DB의 JSONB 포맷에 저장하여 중복 호출 시 대기 시간 없이 즉시 응답하도록 최적화했습니다.

이어서 이러한 범용 채팅 환경을 넘어 연구자만의 개인 맞춤형 공간을 제공하는 'Gem 팩토리'를 소개해 드리겠습니다."

### 🎴 Slide 6. 맞춤형 AI 연구 비서 — Gem 팩토리 파이프라인 (Core Features 03)
* **화면 구성**: 사용자가 정의한 페르소나가 실제 RAG 툴로 변환되어 안전하게 파괴되기까지의 4단계 가로형 워크플로우(Step 1 ~ Step 4).
* **발표 목적**: 보안과 격리가 보장된 개인화 연구 비서(Gem) 기능의 전체 백엔드 생명 주기를 깊이 있게 설명합니다.
* **예상 소요 시간**: 1분 30초

---

#### 🎙️ 발표 대본
"다음은 저희 서비스의 독창적인 기능인 **'맞춤형 AI 연구 비서, Gem 팩토리 파이프라인'**입니다.
사용자는 자신이 원하는 학술 카테고리와 역할극 지침, 즉 페르소나를 부여해 나만의 비서 'Gem'을 만들고, 본인이 가진 PDF 파일을 업로드해 고유한 지식 기반 RAG 대화를 안전하게 나눌 수 있습니다. 이 과정은 총 4단계 파이프라인으로 엄격히 관리됩니다.

* **1단계는 페르소나 바인딩**입니다. 사용자가 RAG 분야와 시스템 프롬프트를 설정하면, 고유한 메타데이터 레코드를 생성하여 `gem` 테이블에 영구 등록합니다.
* **2단계는 PDF 테넌시 격리**입니다. 사용자가 개인 소유의 연구 PDF를 업로드하면, 백그라운드에서 즉시 500자 단위로 청킹한 뒤 벡터화합니다. 이 데이터는 범용 테이블이 아닌 `gem_{gem_id}_files`라는 독립된 전용 컬렉션 테이블에 저장되어, 타 사용자의 접근을 완전히 차단하는 물리적 격리를 이뤄냅니다.
* **3단계는 동적 도구 주입**입니다. 세션이 시작될 때 백엔드는 렉시컬 스코프 내부에서 `gem_id`를 가두는 클로저(Closure) 함수를 동적으로 빌드합니다. 이를 Supervisor Agent의 툴로 실시간 주입하여 해당 Gem이 지정된 개인 컬렉션에서만 검색을 실행하도록 강제 제어합니다.
* **마지막 4단계는 물리 완전 소멸**입니다. 사용자가 연구 비서를 삭제하면 DB Cascade 연쇄 삭제 제약에 따라 메타데이터가 드롭되며, 이와 동시에 물리 격리되어 있던 `gem_{gem_id}_files` 테이블 자체에 `DROP TABLE` 명령을 내려 디스크 저장 공간을 0바이트로 완전 소각 처리합니다. 데이터 잔재로 인한 보안 유출 가능성을 원천 차단한 것입니다.

이렇듯 고도의 기능을 안정적으로 구현하기 위해 저희가 마주했던 기술적 문제들과 그 해결 사례를 다음 장에서 나누겠습니다."

#### 💡 발표 팁
* **'물리적 테넌시 격리'**와 **'클로저를 활용한 동적 도구 주입'**, 그리고 삭제 시 **'DROP TABLE을 통한 물리 완전 소멸'**은 아키텍처 상 매우 훌륭한 보안 설계입니다. 이 단계들을 순차적으로 차분하게 조리 있게 설명하십시오.

### 🎴 Slide 7. 핵심 기술 문제 해결 및 리팩토링 사례 (Troubleshooting)

* **예상 소요 시간**: 1분 15초

#### 🎙️ 발표 대본
"저희가 프로젝트를 진행하며 겪은 핵심 기술 문제와 리팩토링 해결 사례 세 가지를 공유해 드리겠습니다.

* **첫 번째 케이스는 Structured Outputs와 StreamingResponse의 결합 문제**였습니다. LLM이 출처나 추천 질문 등 구조화된 JSON 데이터 형식을 완성하기 전까지는 토큰 단위 실시간 스트리밍이 불가능하여 UX 저하를 유발했습니다.
  * 저희는 이를 해결하기 위해 **이벤트 기반 NDJSON 스트리밍**을 설계했습니다. 실제 답변 본문 텍스트는 즉시 토큰 단위로 실시간 전송하고, 출처 및 추천 질문 같은 구조화 데이터는 스트림 완료 시점에 Pydantic DTO로 엄격히 검증한 뒤 최종 라인으로 병합 전송하여 부드러운 스트리밍 UX와 견고한 API 스키마를 동시에 만족시켰습니다.
* **두 번째 케이스는 10만 건 대용량 데이터 적재 시의 커넥션 락업 문제**였습니다. 대량의 벡터 임베딩을 단일 DB 트랜잭션으로 한 번에 밀어 넣으려다 보니 메모리 누수와 커넥션 락이 걸려 서버가 다운되는 현상이 빈번했습니다.
  * 이를 해결하기 위해 **5,000건 단위로 트랜잭션을 페이징 분할**하여 순차적으로 Commit하고, 매 배치 실행이 끝날 때마다 파이썬 레벨에서 가비지 컬렉터(GC)를 강제 구동하여 메모리를 비워줌으로써, 단 한 번의 중단이나 병목 없이 106,974건의 대용량 데이터를 안전하게 적재하는 데 성공했습니다.
* **세 번째 케이스는 번역 과정에서 발생하는 학술 인용구 왜곡**이었습니다. GPT-4o 번역 기능을 거치는 과정에서 학술 원문의 엄격한 영어 인용구 데이터들이 한글로 무리하게 번역되거나 훼손되어 원문 역추적이 안 되는 심각한 신뢰성 문제가 있었습니다.
  * 저희는 파이썬 백엔드 레이어에서 번역 요청 전 핵심 인용구를 사전 복사(Backup)하여 메모리에 보관해 두고, 한글 번역이 완료된 시점에 복원(Restore)하는 **하이브리드 필터**를 자체 설계하여 인용구 유실률을 0%로 완벽히 방어했습니다.

마지막으로 저희 플랫폼이 가진 한계점과 향후 발전 계획에 대해 말씀드리겠습니다."
